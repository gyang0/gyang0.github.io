For most of high school, I wrote notes by hand. This was pretty time consuming and ended up hurting my hand a lot. The notes themselves weren't very good &ndash; I didn't want to write out entire proofs, for example, so I'd just write the theorems.

In 2025, I started writing notes in LaTeX, mostly to keep up with my Quantum Computing course. Since then, it's grown into a more complicated system. Though I'm still adding to it regularly, I figured I might give an overview of the process.

## Components
My OS is Windows (yes, I know, but I need it for school).

I use VSCode with Markdown to write my notes, then convert that to a PDF with Pandoc. To upload notes to my website, I have a custom Bash script from where I can track all my notes, open a specific project, commit some to my repository, etc.

Within VSCode, I use the <a href="https://marketplace.visualstudio.com/items?itemName=yzhang.markdown-all-in-one" target="_blank">Markdown All in One</a> extension for my LaTeX macros. Additionally, <a href="https://marketplace.visualstudio.com/items?itemName=streetsidesoftware.code-spell-checker" target="_blank">Code Spell Checker</a> has been nice to have, particularly when I'm running off coffee in the dead of night.

Gilles Castel has an <a href="https://castel.dev/" target="_blank">amazing set of posts</a> on LaTeX notetaking. His system uses Vim and Linux, which makes his workflow faster than mine. It was an invaluable resource for growing my own system.

## VSCode & Markdown
As you might have seen above, I write my notes in Markdown + LaTeX. VSCode is my main IDE, and one I'm used to. I tend to like the "natural" syntax of Markdown over LaTeX for taking notes. The built-in PDF preview is helpful as well.

The `setup.tex` contains all my macros and packages. All notes go into the `notes` folder. I split notes by week (for lectures) or by chapter (for books). All images go into `img`. When I'm done, I run `compile.sh`, which uses Pandoc to generate a LaTeX PDF from my Markdown file. Then the `output/TOTAL.pdf` is overwritten with that new PDF.

`compile.sh` itself is a simple script that calls Pandoc, but I added some commands to modify the look.

```[bash]
#!/bin/bash

TITLE="Notes for Complex Analysis"
YEAR="2026"

# Additive total
total_file_str="% $TITLE"$'\n'"% Made by Gene"$'\n'"% $YEAR"$'\n'

echo "Combining MD files..."

# https://stackoverflow.com/a/43661580
files=`ls ./notes/*.md`
for file in $files
do
    # Current file
    file_str=$(<$file)

    # Replace $$\begin{...} ... \end{...}$$ with \begin{...} ... \end{...}
    # works for gather* and align*
    # VSCode markdown needs the $$, but incompatible with Pandoc.
    file_str="${file_str//\$\$\\begin\{gather\*/\\begin\{gather\*}"
    file_str="${file_str//\$\$\\begin\{align\*/\\begin\{align\*}"
    file_str="${file_str//\\end\{gather\*\}\$\$/\\end\{gather\*\}}"
    file_str="${file_str//\\end\{align\*\}\$\$/\\end\{align\*\}}"

    # For complete set of notes
    echo "    $file completed."
    total_file_str+="$file_str"$'\n'$'\n'
done 

# All notes combined
echo "Generating PDF..."
echo "$total_file_str" | pandoc -s -o "output/TOTAL.pdf" --katex --toc -N -f markdown-implicit_figures -V geometry:"margin=1in" -V fontsize=11pt --resource-path=./notes --include-in-header "_setup/setup.tex"
echo "Complete."
```

<br>

The way VSCode handles the LaTeX math environment is kind of finnicky. Though there's probably a better way to do it, I just hardcoded the `gather` and `align` environments, which are what I use 99% of the time anyway.

An example document generated by this script can be found <a href="/data/documents/20260218_complex_analysis_notes.pdf">here</a>. I use 1 inch margins with an 11pt XCharter font, which is highly legible even at small sizes. But the default LaTeX configurations work very well too.

## Macros
To speed up my notetaking, I use KaTeX macros, via the "Markdown All in One" extension. I add them to VSCode's `settings.json` file. They're also added to my project's `setup.tex`, so that Pandoc can access them.
<figure>
    <img style="width:500px" src="/images/activity/2026/20260309_notetaking_macros.png">
    <figcaption>Example macros</figcaption>
</figure>

Additionally, I have 3 snippets for VSCode: `\beg` (for `\begin{...}\end{...}`), `\box` for `\parbox`, and `\fig` for Markdown figures. These don't seem like much, but they save a ton of time &ndash; consider how much you have to type `\begin{gather*}\end{gather*}` in a typical document! Instead, I can just type `\beg` and tab to autocomplete.

## Note-Managing App
I save most of my notes to my site (the thing you're reading right now). This means that every time I edit my notes, I have to re-upload them to my local site, commit to GitHub, etc. It's time-consuming, which is why I recently made a Bash script to manage my notes.
<figure>
    <img style="width:600px" src="/images/activity/2026/20260309_notes_manager.png">
    <figcaption>Notes Manager</figcaption>
</figure>

I use <a href="https://cmder.app/" target="_blank">cmder</a> (a ConEmu wrapper) with the <a href="https://github.com/PandaTheme/Panda-Theme-Cmder" target="_blank">Panda color theme</a>. The program is just an infinite loop that does things based on user input: commit projects, edit project details, register a new project, etc. The code is just one bash file that can be found <a href="https://github.com/gyang0/Notes-Manager" target="_blank">here</a>, but it's tailored to my system, so probably not as helpful for others.

I pinned `main.sh` to my taskbar by creating a new desktop shortcut with target
```[bash]
C:\Users\user\cmder\vendor\conemu-maximus5\ConEmu.exe -run cmd /k "bash main.sh"
```

<br>

Then whenever I want to edit my notes, I just click that icon, open the necessary project with `open # -v`, and run `compile.sh` to generate the new PDF. Afterwards, I commit to my site with `commit #`. Much faster.

Other than easy site updates, I find this helpful to quickly open a project PDF or VSCode environment. Instead of digging through my files, I just do `open # -pv` to open both the PDF and VSCode.

## Future
I'm quite happy with my system for the time being, but there's always room for improvement. For one thing, I don't have a good system for drawing my own figures. Gilles Castel uses <a href="https://apps.microsoft.com/detail/9pd9bhglfc7h" target="_blank">Inkscape</a>, which I may try later. Furthermore, typing in VSCode (even with macros) is likely slower than Vim. Though I've mostly put off learning Vim, maybe this is a good time to start.

I expect to use this system a lot next year in college.
